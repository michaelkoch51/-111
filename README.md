## 🛠 Задание 1. Стратегия резервного копирования

В финансовом секторе требования к сохранности данных (RPO) и времени восстановления (RTO) критически высоки. Ниже представлены оптимальные архитектурные решения под конкретные бизнес-кейсы.

### 📋 Сводная таблица решений


| Кейс | Бизнес-требование | Техническое решение | Описание и риски |
| :--- | :--- | :--- | :--- |
| **1.1** | Восстановление в полном объёме за предыдущий день | **Full Backup** (Полный бэкап) | Раз в сутки (ночью) создается полная копия БД. Риск: потеря данных за текущий рабочий день при сбое. |
| **1.2** | Восстановление за час до предполагаемой поломки | **Incremental/Differential Backup** или **PITR** | Сочетание ночного полного бэкапа и ежечасного копирования изменений, либо непрерывное архивирование журналов (WAL/Binlog). |
| **1.3\*** | Моментальное переключение на работающую БД | **High Availability (HA) Cluster** | Репликация данных на Slave-сервер с автоматическим Failover (переключением трафика) за секунды. |

## 🐘 Задание 2. Работа с PostgreSQL

### 2.1. Резервирование и восстановление через утилиты `pg_dump` и `pg_restore`

Для создания бэкапов используется кастомный сжатый формат (`-F c`), который позволяет гибко восстанавливать отдельные таблицы или всю базу целиком.

<details>
<summary>💻 Посмотреть примеры команд PostgreSQL (CLI)</summary>

**Создание резервной копии (Бэкап):**
```bash
pg_dump -U username -h localhost -F c -b -v -f /path/to/backup/my_db_backup.dump my_database
```

**Восстановление базы данных:**
```bash
pg_restore -U username -h localhost -d my_database -v /path/to/backup/my_db_backup.dump
```

*Разбор ключевых флагов:*
* `-F c` — кастомный (custom) архивный формат (сжатый).
* `-b` — включать большие объекты (blobs).
* `-v` — подробный вывод процесса (verbose).
</details>

### 2.1.* Автоматизация процесса
В промышленной эксплуатации ручной запуск недопустим. Автоматизация реализуется двумя путями:
1. **Базовый (cron + bash):** Скрипт автоматизации с использованием файла аутентификации `~/.pgpass` для безопасного беспарольного доступа.
2. **Enterprise-подход:** Использование специализированного ПО оркестрации бэкапов, такого как **pgBackRest**, **Barman** или **pg_probackup** (поддерживают инкременты на блочном уровне и выгрузку в S3-хранилища).

## 🐬 Задание 3. Работа с MySQL

### 3.1. Инкрементное резервное копирование

В экосистеме MySQL инкрементное резервное копирование реализуется двумя основными путями: на логическом уровне через **бинарные логи (Binary Logs)** или на физическом уровне с помощью утилиты **Percona XtraBackup**.

<details>
<summary>💻 Посмотреть примеры команд инкрементного копирования MySQL</summary>

#### Подход №1: Использование штатных бинарных логов (Point-in-Time Recovery)
Сначала создается полный базовый слепок данных, а текущий журнал транзакций закрывается:
```bash
mysqldump -u root -p --all-databases --flush-logs --master-data=2 > /backup/full_base.sql
```
Для фиксации последующих инкрементов (изменений) выполняется принудительная ротация логов. Каждый новый файл вида `binlog.00000X` в директории MySQL будет являться инкрементным бэкапом:
```bash
mysqladmin -u root -p flush-logs
```

#### Подход №2: Использование утилиты Percona XtraBackup (Промышленный стандарт)
Создание физического инкремента на основе изменений в страницах данных относительно ранее созданной полной копии:
```bash
xtrabackup --backup --target-dir=/backup/inc_1 --incremental-basedir=/backup/full_base
```
</details>

### 3.1.* Преимущества репликации перед классическим бэкапом

Использование реплики (Slave/Replica) предоставляет финансовой компании критически важные преимущества в рамках обеспечения непрерывности бизнеса (Business Continuity):

* ⏱️ **Минимальный RTO (Recovery Time Objective)**  
  При отказе основного сервера переключение на реплику занимает секунды или минуты. Разворачивание традиционного терабайтного бэкапа из архива может занять часы, что приведет к огромным финансовым потерям из-за простоя.
* 📉 **Минимальный RPO (Recovery Point Objective)**  
  Реплика получает изменения практически мгновенно в режиме реального времени. Обычный бэкап ограничен частотой своего запуска (например, раз в час), из-за чего при аварии можно потерять данные за целый час работы.
* 🚀 **Снятие нагрузки с Production-сервера**  
  Тяжелые аналитические отчеты (сложные `SELECT`-запросы) и сам процесс снятия классических бэкапов можно перенести на реплику. Это позволяет освободить ресурсы основного сервера для обслуживания транзакций клиентов.
* 🌍 **Географическое распределение рисков**  
  Реплику можно разместить в изолированном, удаленном дата-центре. Это гарантирует выживаемость данных даже в случае полной техногенной аварии или обесточивания основного ЦОД.
